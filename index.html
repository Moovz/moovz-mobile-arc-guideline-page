Mobile Architecture Guidelines

Guia curto para novas features e refactors no mobile. Cobre duas coisas: as regras de processo de PR — o que o CI valida automaticamente e bloqueia — e as regras de arquitetura, que ficam a cargo do review humano.

Nome de branch CI bloqueia

Roda em todo PR aberto contra develop.

Padrão exigido

(feature|fix|refactor|chore|hotfix)/algo-com-traco

Válidos: feature/T-500-fix-login · fix/bug-checkin · refactor/unify-app-bars

Mensagem de commit CI bloqueia

Padrão exigido

tipo: descrição

Tipos aceitos (case-insensitive):

feat fix docs style refactor perf test build ci chore revert hotfix

  • título precisa ter pelo menos 5 caracteres;
  • sem palavrão (lista em .github/scripts/bad_words.txt);
  • commits de merge (Merge ...) são ignorados.
Hook local opcional: hooks/commit-msg (ativado via hooks/setup.sh) aplica a mesma regra antes do commit ser criado, pra feedback mais rápido. Não substitui o check do CI — é só um atalho.

Estrutura de arquivos em lib/features/** CI bloqueia

Só vale pra arquivos .dart novos — arquivo existente não é revalidado. .md e outras extensões ficam livres, inclusive na raiz da feature.

Sufixo do arquivo novoPrecisa estar dentro de
_model.dart, _repository.dartdata/
_service.dartservice/
_viewmodel.dartui/
_view.dartsem regra de posição
qualquer outro (utils, constants, widget solto)data/, service/, ui/, widgets/ ou components/ (sinônimos)

Exemplo de estrutura completa

data/ é o nome oficial para Model + Repository (não repository/ solto) — segue o mesmo vocabulário usado nos guias de arquitetura Android/Flutter ("data layer"), e dá um lugar natural tanto pro Model quanto pro Repository sem precisar inventar outra pasta de topo.

Isso vale só pra arquivo novo: features antigas com estrutura diferente não precisam ser migradas retroativamente.

Direção de import entre camadas CI bloqueia

Mesma lógica do Fluxo de camadas, aplicada automaticamente em arquivo novo.

  • arquivo novo _view.dart ou _viewmodel.dart não pode importar nenhum *_repository.dart;
  • arquivo novo _service.dart não pode importar nenhum *_viewmodel.dart.

Se o CI reprovar por isso, o import precisa ser removido — normalmente é sinal de que falta uma camada no meio (ex: UI chamando Repository direto em vez de passar pelo ViewModel/Service).

Fluxo de camadas

UI → ViewModel → Service → Repository
  • UI não deve importar ou chamar Repository.
  • ViewModel não deve chamar Repository diretamente.
  • Service concentra regra de negócio e orquestra chamadas de Repository.
  • Repository fica responsável por acesso a dados externos: Firestore, Storage, APIs, cache, etc.

CI  as duas primeiras regras (UI/ViewModel → Service e Repository) já são verificadas pelo CI pra arquivo novo. As demais dependem de review humano.

Injeção de dependências

Services e ViewModels devem ser registrados no getIt:

  • dependências compartilhadas/stateless: registerLazySingleton;
  • ViewModels de tela: registerFactory;
  • objetos com ciclo de vida da tela devem ser descartados pela tela quando aplicável.
Preferir 
late final ExampleViewModel _viewModel;

@override
void initState() {
  super.initState();
  _viewModel = locator<ExampleViewModel>();
}

@override
void dispose() {
  _viewModel.dispose();
  super.dispose();
}
Evitar 
_viewModel = ExampleViewModel(
  service: locator<ExampleService>(),
  userId: AuthService.instance.userState.id,
);

A tela não deve montar o grafo de dependências nem carregar parâmetros de regra de negócio para o ViewModel.

UI

Deve
  • renderizar estado;
  • disparar ações simples para o ViewModel;
  • navegar quando fizer sentido para a camada de apresentação;
  • manter baixo conhecimento sobre Services e nenhum sobre Repositories.
Não deve
  • montar Services manualmente;
  • resolver Repository;
  • decidir regra de negócio;
  • guardar estado que pertence ao ViewModel ou Service.

ViewModel

Deve
  • expor estado pronto para a tela;
  • chamar Services;
  • transformar dados em propriedades de UI quando necessário;
  • controlar loading, erro e dispose de subscriptions.
Não deve
  • acessar Repository diretamente;
  • receber muitos parâmetros da tela;
  • depender de widgets, BuildContext ou navegação;
  • guardar constantes de regra de negócio que o Service precisa usar.

Service

  • conter regra de negócio;
  • chamar Repository;
  • resolver dados necessários ao fluxo;
  • retornar objetos/estados claros para o ViewModel.

Evitar Service singleton guardando estado temporário de tela:

Evitar 
String? currentUserId;
SomeMode? currentMode;
SomeModel? currentItem;
Preferir 
class SummaryState {
  final String userId;
  final SummaryMode mode;
  final SomeModel? lastItem;
}

Esse tipo de estado pode vazar entre aberturas de tela, usuários ou fluxos diferentes — prefira retornar um objeto de resultado.

Widgets e arquivos de tela

Evitar arquivos monolíticos com tela, header, cards, listas e widgets auxiliares no mesmo arquivo. Preferir dividir por responsabilidade:

feature/ ui/ example_screen.dart widgets/ example_header.dart example_body.dart example_card.dart

Widgets pequenos devem receber apenas os dados que usam:

Preferir 
StatsCard(
  total: viewModel.total,
  average: viewModel.average,
)
Evitar 
StatsCard(viewModel: viewModel)

Debug e testes manuais

Antes de abrir PR ou pedir review:

  • remover flags temporárias;
  • não deixar strings como 'null' simulando null;
  • remover comentários de teste manual que instruem alterar código;
  • garantir que fluxos de erro não sejam tratados como lista vazia/sucesso silencioso.

Checklist de review

Antes do merge, validar:

  • a tela não importa Repository CI — revisar mesmo assim em arquivo modificado;
  • ViewModel vem do getIt quando seguir o padrão da feature;
  • ViewModel depende de Service, não de Repository CI;
  • Service não depende de ViewModel CI;
  • singletons não guardam estado temporário de tela;
  • arquivos grandes foram quebrados em widgets/componentes menores;
  • widgets visuais recebem dados específicos, não ViewModel inteiro sem necessidade;
  • chamadas async que usam context verificam context.mounted;
  • erros de Repository/Service não são escondidos como estado vazio.
